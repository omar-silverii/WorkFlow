## 5. NOTA_CREDITO_ELECTRONICA_AR / NC_AR

En esta sesión se incorporó y validó el nuevo DocTipo para nota de crédito electrónica argentina.

Decisión funcional:

```text
Codigo = NOTA_CREDITO_ELECTRONICA_AR
Nombre = Nota de crédito electrónica AFIP
ContextPrefix = notaCredito
MotorExtraccion = NC_AR
```

Se creó extractor propio:

```text
DocumentProcessing/NotaCreditoElectronicaArExtractor.cs
```

Se decidió separar motor y DocTipo porque las notas de crédito son comprobantes propios y no conviene mezclar toda la casuística dentro de `FACTURA_AR`.

### Decisiones de datos

Los importes se guardan positivos tal como vienen en el PDF.

Además se agregan campos de comportamiento:

```text

biz.notaCredito.signo = -1
biz.notaCredito.tipoMovimiento = CREDITO
```

Se agregan datos del comprobante asociado:

```text

biz.notaCredito.comprobanteAsociado.numero
biz.notaCredito.comprobanteAsociado.letra
biz.notaCredito.comprobanteAsociado.puntoVenta
biz.notaCredito.comprobanteAsociado.numeroComprobante
```

### Script SQL

Al correr el alta del nuevo DocTipo apareció error:

```text
CHECK constraint CK_WF_DocTipo_MotorExtraccion
```

Causa:

La columna `MotorExtraccion` ya existía, pero el constraint solo permitía valores anteriores, por ejemplo:

```text
REGLAS
FACTURA_AR
```

Solución aplicada/recomendada:

```sql
ALTER TABLE dbo.WF_DocTipo
DROP CONSTRAINT CK_WF_DocTipo_MotorExtraccion;
GO

ALTER TABLE dbo.WF_DocTipo WITH CHECK
ADD CONSTRAINT CK_WF_DocTipo_MotorExtraccion
CHECK (MotorExtraccion IN ('REGLAS', 'FACTURA_AR', 'NC_AR'));
GO

ALTER TABLE dbo.WF_DocTipo
CHECK CONSTRAINT CK_WF_DocTipo_MotorExtraccion;
GO
```

Luego se pudo dar de alta el DocTipo con `NC_AR`.

No debe pegarse JSON ni `"path":`.

## 7. Nueva línea grande — Asistente IA

Omar planteó una preocupación muy importante:

Después de estar más de un mes sin tocar el proyecto, al volver al editor se sintió paralizado para crear grafos. No sabía por dónde empezar ni qué grafo hacía qué cosa.

Conclusión:

El problema no es técnico sino de experiencia de creación.

El usuario final podría sentir:

```text
Esto es muy complicado, no lo quiero usar.
```

Se conversó primero sobre plantillas y asistente guiado, pero Omar aclaró que NO quiere una solución simple del tipo:

```text
Aprobación simple → grafo terminado
```

Quiere algo más interactivo.

### 7.1 Idea deseada

Crear una experiencia donde el usuario vaya diciendo lo que quiere hacer y el sistema lo ayude progresivamente.

Ejemplos:

```text
Quiero validar.
Quiero enviar a fulano o a tal rol.
Quiero mandar un correo.
Quiero guardar en base.
Quiero cargar una factura.
Quiero pedir aprobación si supera cierto importe.
```

Y el sistema debería:

* interpretar la intención
* proponer nodos
* pedir datos faltantes
* mostrar usuarios/roles válidos
* dibujar nodos en el canvas
* conectarlos progresivamente
* validar lo generado
* enseñar mientras construye

Omar quiere algo más cercano a un copiloto/asistente de construcción de workflows que a plantillas cerradas.

### 7.2 Decisión de nombre

Nombre elegido:

```text
Asistente IA
```

### 7.3 Decisión estratégica

Omar propuso pensar en grande: Workflow Studio con asistencia de IA sería mucho más vendible.

Se acordó animarse a una línea grande, pero con arquitectura seria.

Principio central:

```text
La IA interpreta y propone.
Workflow Studio valida y aplica.
```

La IA no debe tocar SQL, código C# ni modificar archivos directamente.

Debe devolver una estructura controlada que el sistema valida.

### 7.4 Decisión crítica: sin internet / intranet

Omar aclaró algo fundamental:

```text
No podemos salir a internet.
Debe pensarse desde el inicio para intranet.
```

Por lo tanto:

* no diseñar la primera versión dependiendo de OpenAI API externa
* no hacer algo que funcione en desarrollo pero falle al pasar a intranet
* pensar desde el arranque en un Asistente IA offline/local o intranet-compatible
* aceptar que cueste más, pero hacerlo con el escenario real de uso

Esta decisión es clave y debe respetarse.

### 7.5 Arquitectura conceptual deseada

Flujo conceptual:

```text
Usuario escribe intención
        ↓
Asistente IA local/intranet interpreta
        ↓
Devuelve plan estructurado
        ↓
Validador interno revisa coherencia
        ↓
Compilador de acciones transforma plan en nodos/aristas
        ↓
Canvas dibuja el grafo
        ↓
Usuario confirma/corrige/sigue agregando
```

### 7.6 La IA no debe inventar

El Asistente IA solo puede trabajar con el conocimiento del producto:

* nodos existentes
* parámetros válidos
* DocTipos activos
* roles disponibles
* usuarios disponibles
* variables disponibles
* outputs conocidos
* reglas de conexión
* handlers implementados
* convenciones `wf.*`, `biz.*`, `input.*`

Si el usuario pide algo inexistente, el asistente debe decirlo y ofrecer una alternativa posible con nodos reales.

### 7.7 Primera línea de diseño futuro

Antes de código, diseñar un documento:

```text
Diseño del Asistente IA de Workflow Studio v1
```

Debe definir:

* qué puede pedir el usuario

* qué nodos puede crear

* cómo se representa la respuesta
* cómo se pide información faltante
* cómo se aplica al canvas
* cómo se revierte/cancela una propuesta
* cómo se explica lo que creó
* cómo funciona sin internet
